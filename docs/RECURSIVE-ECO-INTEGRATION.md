# How recursive-tarot connects to recursive.eco

This repo (**recursive-tarot**) is the *open* half of the project: public, static, GitHub-Pages
served, data licensed CC-BY-SA-4.0. **recursive.eco** is the *app* half: private (Next.js +
Supabase + Cloudflare R2). They share **one thing — the grammar JSON format** — and pass grammars
back and forth across the GitHub boundary. We share the format, not the platform.

This doc is the single reference for that boundary: where a grammar lives in each system, which way
data flows, who owns which fields, and how to keep a sync from clobbering repo work.

---

## 1. Where a grammar lives in each system

| | recursive-tarot (this repo) | recursive.eco (the app) |
|---|---|---|
| Storage | `tarot/<slug>/grammar.json` (a file) | `user_documents.document_data` (JSONB, one row) |
| Identity | the folder **slug** (e.g. `visconti-sforza-tarot`) | a **UUID** (e.g. `6dc092b0-…`) |
| Source of truth | git history | the database row |

The slug → UUID map is **`tarot/_eco_ids.json`** (also lists which decks are `is_public` on the app,
and the link conventions: `?deckId=<id>` to open/read, `?forkId=<id>` to fork/edit).

---

## 2. The two directions of flow

```
                 reads (import / Cast)
   recursive.eco  ◀───────────────────────  recursive-tarot @ main
   (Supabase row)                            (tarot/<slug>/grammar.json)
                 ───────────────────────▶
                 writes (channel sync PR)
```

### App → reads the repo (import / Cast)
recursive.eco pulls a deck straight from raw GitHub:
```
https://raw.githubusercontent.com/PlayfulProcess/recursive-tarot/main/tarot/<slug>/grammar.json
```
**Note the branch: `main`.** The app reads `main`, never `dev`. So a repo grammar only reaches the
app once it's on `main`.

### App → writes the repo (channel sync / "Resolve all drifts")
The **`app/recursive-eco` GitHub App** opens a pull request to this repo containing regenerated
`grammar.json` files — one per grammar that has *drifted* (the app's copy differs from the repo's).
It's triggered from the app's channel-sync panel. The PR is reviewed and merged to **`main`**.

The app serializes a grammar with `document-data-to-grammar-json.ts`, which is **verbatim**
(`{ ...documentData, backlinks }`) — it does *not* drop fields it "doesn't model." That matters for
§4 below.

---

## 3. Branches: `dev` vs `main`

- **`dev`** — the **live static site** (`tarot.recursive.eco`) and the active working branch. All
  building happens here. CI rebuilds the meta-grammar on `dev`.
- **`main`** — the branch the **app reads from and writes to**. The app's import path is pinned to
  `main`; the app's sync PRs target `main`.

So: **merge `dev` → `main`** to publish work to the app's read path. The app's write-backs (community
edits, "Resolve all drifts") land on `main`, sometimes ahead of `dev` — always reconcile rather than
discard (e.g. the App's `charles-vi` community edit has lived only on `main`). Use a normal merge
commit; don't force one branch over the other.

---

## 4. Ownership — who's the source of truth for what

The sync only stays lossless if both sides agree on ownership. Three buckets:

### A. App-owned fields (the database is source of truth)
The app's copy wins for these on a write-back:
`items`, `name`, `description`, the AI personality prompt, publish state, `default_view`,
`open_to_community`, and the app's roundtrip URLs (`_recursive_eco_url`, `_recursive_eco_edit_url`,
`_forked_from_url`, `_backup_repo_url`, `_detected_categories`).

### B. Repo-only fields (repo scripts own these; the app never ingests them)
These are added by this repo's own tooling **after** a grammar was last imported into the app, so the
app's snapshot doesn't contain them. A naïve write-back (replace the file with the app's snapshot)
**deletes** them. They must survive:
- **`_grammar_commons`** — the CC-BY-SA license + attribution block. **On all 38 grammars.** Losing
  this strips the licence.
- **`_image_usage`** — `audit_image_usage.py` output (which page each image feeds).
- **`lineages` / `roots` / `shelves` / `worldview` / `print_readiness`** — the "many-lenses" /
  genealogy / print scaffolding (e.g. on `golden-dawn-book-t-tarot`).
- **`provenance`**, the `_github_*` URLs, and `_enriched_from` / `_research`.
- The **cross-link pills** on items (`source_deck` / `source_item_id` / `deck`) and **Research-note**
  sections with `[@citation]` keys (added by `enrich_cards_from_research.py` / `build_people_grammar.py`).

### C. Repo-owned / generated grammars (the repo is the *whole* source of truth)
These are *generated* in the repo from other sources — the app must **not** push its copy back at all
(doing so reverts or clobbers them). They're stamped `_generated: true`:

| grammar | generated by |
|---|---|
| `all-decks-many-lenses` | `build_meta_grammar.py` (the meta-grammar) |
| `people-of-tarot` | `build_people_grammar.py` (also `_source_of_truth: "research/people/*.md"`) |
| `how-to-contribute` | `build_contribute_grammar.py` |
| `history-of-tarot-course` | `course_to_grammar.py` (an MDX course → grammar; see [the course note](#6-the-course-as-a-grammar)) |

> The app's sync should **skip any grammar with `_generated: true`** (and/or `_source_of_truth` set).
> Today the repo reliably stamps `_generated: true` on the four above; it does **not** yet stamp
> `_source_of_truth: 'repo'` — if the app keys its skip on that value, the generators need to write it
> (a one-line addition per generator).

## 5. The drift problem and the write-back fix

**Symptom (PR #28, Jun 2026):** "Resolve all drifts" opened a PR of +26k/−27k lines across 27
grammars. Card content was fully preserved, but it would have **dropped repo-only fields** (§4B) on
several decks (clearest: `golden-dawn` losing `_grammar_commons`, `lineages`, `print_readiness`, …)
and **reverted** the course grammar — and the diff was unreviewable (≈99% key-reorder noise).

**Root cause (not the serializer):** the serializer is verbatim. The loss comes from the write-back
**replacing the repo file with the app's *stale snapshot*** — the app's copy predates the repo
scripts that added `_grammar_commons` etc., so pushing stale-over-fresh clobbers repo-only keys, and
the app's key order explodes the diff.

**Fix (on the app side — merge-onto-repo at write-back):** before committing each file, fetch the
*current* repo `grammar.json` and emit `{ ...repoFile, ...appDocumentData, backlinks }`. App-owned
keys win; repo-only top-level keys survive; **starting from the repo object also preserves the repo's
key order**, so the diff collapses to real changes. Plus: **skip `_generated` / repo-owned grammars**,
and run **drift detection on an app-relevant projection** so repo-only fields stop perpetually
flagging drift.

> Known edge of the spread approach (worth a note in the app): it's a **shallow** top-level merge, so
> a repo enrichment nested *inside* an app-owned object (e.g. a key the repo adds to the top-level
> `metadata` object) is still overwritten by the app's version of that object; and a field the app
> *intentionally deleted* at the top level will be **resurrected** from the repo file. For the fields
> we have today (all top-level, repo-only) the shallow merge is correct; a field-ownership allowlist
> would be the precise version if nested ownership ever appears.

**After the fix lands on the app:** re-run "Resolve all" → the regenerated PR preserves licence +
enrichment + key order, skips the course, and is small and reviewable.

## 6. The course as a grammar

`history-of-tarot-course` is a normal grammar **generated by `course_to_grammar.py`** from
`course/history-of-tarot.mdx`: each `##` section becomes a lesson; the card galleries become embeds
(see `viewers/course-embeds.js`). It renders identically to the MDX course-viewer via
`viewers/grammar-course.html`. Because it's a **grammar**, the app's **Library Assistant could read
its content** — today the assistant can open a course but not see inside it. As repo-generated
content it's `_generated: true` and must be **skipped** by app→repo sync (the repo is its source of
truth; the app should *pull*, not push).

---

## 7. The cross-link pill (the one cross-grammar nav mechanism)

The viewers render an "Open in [deck] →" pill when an item has `metadata.source_deck` +
`source_item_id` + `deck`. This is the **only** cross-grammar link mechanism — used for meta→deck,
deck→people, deck→related. See `CLAUDE.md` ("The one cross-link pattern"). Don't invent another link
field, and remember it's a §4B repo-only enrichment the sync must preserve.

---

## 8. The recursive.eco grammar MCP

From Claude Code you can build/publish/cast against the app directly via the **recursive.eco grammar
MCP** (server `6b361191-1fb9-4ce8-bfa5-928a3833b8a0`): `list_grammars`, `get_grammar`,
`create_grammar`, `add_items`, `set_grammar_visibility`, `cast`, image/audio tools, etc. This is how
repo grammars get published to the app and how `_eco_ids.json` was sourced. (claude.ai needs the
connector's tools set to **Always allow**, or its approval gate blocks the call — Claude Code is not
affected.)

## Quick reference

- **App reads:** `…/recursive-tarot/main/tarot/<slug>/grammar.json` (branch = `main`).
- **App writes:** PRs from `app/recursive-eco` → `main` ("Resolve all drifts").
- **Publish to app:** merge `dev` → `main`.
- **Never let sync push:** `_generated: true` grammars (meta, people, contribute, course).
- **Never let sync drop:** `_grammar_commons`, `_image_usage`, `lineages`/`print_readiness`/…,
  cross-link pills, Research notes.
- **Slug ↔ UUID:** `tarot/_eco_ids.json`.
